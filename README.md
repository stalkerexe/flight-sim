# Explorer — Post-Apocalyptic Flight Sim (Этап 8: коллизия с рельефом)

Проверено: `tsc --noEmit` (strict) и `vite build` проходят чисто.

## Коллизия с рельефом (пункт 1 из плана)

Самолёт раньше свободно пролетал сквозь горы и землю — теперь на каждом
кадре после интеграции позиции `Aircraft.ts` запрашивает высоту земли под
собой через ту же чистую функцию `TerrainNoise.getHeight()`, что строит
видимую геометрию (с той же частотой Найквиста, что у ближнего LOD0 — самолёт
"натыкается" ровно на тот рельеф, который видит игрок, не на приближение).

При контакте (`высота самолёта < высота земли + COLLISION_MARGIN`):
скорость обнуляется, самолёт помечается как `crashed`, физика управления
замораживается (обрабатывается в самом `Aircraft.update()` — ранний return),
на экране появляется сообщение "Столкновение с рельефом — нажмите R для
перезапуска". По `R` (`AircraftController`) самолёт сбрасывается в
стартовую точку с нулевой скоростью и газом по умолчанию.

**Известное упрощение**: любое касание рельефа = крушение, нет разделения
на "жёсткая посадка на подходящей скорости/угле" vs "врезался" — полноценная
посадочная механика (шасси, скорость приземления, угол) осталась бы для
будущего этапа миссий/аэродромов. Вода тоже пока не проверяется на
коллизию — можно пролететь сквозь неё низко над океаном без последствий.

## Правка баланса скорости (по фидбэку "летит очень далеко, газ непонятен")

Равновесная скорость (тяга = сопротивление) оказалась нереалистично огромной:

```
                          было         стало
throttle=0.5 (~старт):   726 км/ч   →   162 км/ч
throttle=1.0 (полный):  1320 км/ч   →   324 км/ч
```

На скорости 700+ км/ч любое изменение газа тонуло на фоне и так огромной
скорости — отсюда "непонятно, как работает Shift/Ctrl". Пересчитал
`MAX_THRUST_ACCEL` и `DRAG_COEFF` под реалистичную скорость лёгкого
самолёта, плюс добавил жёсткий потолок `MAX_SPEED = 100 м/с` в
`Aircraft.ts` — независимо от баланса тяги/сопротивления (например, при
затяжном пикировании гравитация может разогнать сильнее равновесной
горизонтальной скорости).

Также добавил в HTML/HUD (`index.html`) постоянно видимую подсказку по
управлению — раньше её не было нигде, кроме README, из-за чего Shift/Ctrl
было неоткуда узнать, кроме как спросить.

## Запуск

**Windows — проще всего:** дважды кликните `start.bat`. Он сам проверит
Node.js, поставит зависимости при первом запуске (`npm install`, только
если папки `node_modules` ещё нет) и поднимет dev-сервер, открыв браузер
через пару секунд. Чтобы остановить — закройте окно консоли или `Ctrl+C`.

Для production-сборки (папка `dist/`, без dev-сервера) — `build.bat`.

**Вручную (любая ОС):**
```bash
npm install
npm run dev        # http://localhost:5173
npm run build       # production build (dist/)
```

## Управление

Теперь это управляемый самолёт, а не свободная камера:

| Клавиша         | Действие                        |
|-----------------|----------------------------------|
| `W` / `S`       | тангаж: нос вниз / вверх         |
| `A` / `D`       | крен влево / вправо              |
| `Q` / `E`       | рысканье (руль направления)      |
| `Shift` (держать) | увеличить газ                  |
| `Ctrl` (держать)  | уменьшить газ                  |

Камера — вид от третьего лица позади самолёта, следует за ним со сглаживанием.

## Что изменилось в этом проходе (по фидбэку с реального запуска)

**1. "Много зелёных гор, мало суши" — амплитуда, не только форма.**
Баг был глубже, чем казалось после прошлой правки площади суши/гор: и
"равнины", и "горы" использовали ОДНУ И ТУ ЖЕ амплитуду высоты
(`WORLD.MAX_TERRAIN_HEIGHT`) — маска гор влияла только на ФОРМУ (ridged
хребты vs обычный fBm), а не на ВЫСОТУ. То есть "равнины" технически не
были ridged-горами, но поднимались до той же высоты, что и настоящие горы —
визуально всё выглядело как "горы разной формы". Исправлено в `Noise.ts`:
теперь амплитуда явно интерполируется между `NOISE.FLAT_AMPLITUDE` (0.16 —
равнины реально плоские) и `NOISE.MOUNTAIN_AMPLITUDE` (1.0) по той же маске
гор, что определяет форму.

**2. "Тормозит на слабых ПК" — понижен базовый уровень + добавлено
адаптивное качество.**
Раньше был один фиксированный набор настроек (129×129 сетка чанка, shadow
map 2048, дальность обзора 48 км) без какой-либо адаптации под слабое
железо. Теперь:
- Базовое разрешение чанка снижено 129→97 (~57% вершин от прежнего).
- Shadow map снижен 2048→1024.
- Добавлен `systems/QualityManager.ts`: измеряет скользящее среднее FPS
  (с cooldown 5 сек и гистерезисом, чтобы не дёргаться на разовых просадках)
  и переключает пресет качества между `low` / `medium` / `high`
  (`config/Constants.ts → QUALITY_TIERS`) — разные радиусы LOD-колец, тени
  вкл/выкл, cap на pixel ratio. Стартует с `medium` (не с самого тяжёлого
  пресета) — безопаснее по умолчанию с учётом явных жалоб на тормоза.
- При downgrade `ChunkManager.setRingRadiusMeters()` немедленно
  пересчитывает набор чанков под новый (меньший) радиус обзора, а
  `Renderer.applyQuality()` — pixel ratio и глобальный флаг теней.
- HUD показывает текущий пресет качества (`Quality: low/medium/high`), FPS
  и число draw call'ов — видно, что реально происходит, а не только "тормозит".

**3. "Самолёта нет" — есть теперь.**
`aircraft/Aircraft.ts` — управляемое летающее тело: простая процедурная
низкополигональная геометрия (капсула фюзеляжа + крылья + хвост из
THREE-примитивов, без внешних GLTF-ассетов) и физика полёта — тяга вдоль
носа, гравитация, упрощённая подъёмная сила (растёт со скоростью вдоль
продольной оси, отдалённо имитируя просадку на малой скорости), демпфирование
скорости. `systems/AircraftController.ts` читает клавиатуру, `systems/ChaseCamera.ts`
— плавная камера от третьего лица. Самолёт (а не камера) теперь якорь
FloatingOrigin/ChunkManager — мир стримится вокруг него.

**ЧЕСТНО про физику самолёта**: это НЕ полная аэродинамика. Нет угла атаки,
нет срыва потока по настоящей физике, нет отдельных control surfaces со своей
эффективностью, нет ground effect. Это приближение, которое уже даёт ощущение
массы/инерции/тяги вместо "камеры-духа" — полноценная модель (`physics/`,
lift = f(AoA), drag polar, отдельные рулевые поверхности) остаётся отдельным
следующим этапом, не смешанным с геометрией самолёта, как сейчас в одном файле.

## Структура (обновлённые/новые файлы)

```
src/
  aircraft/Aircraft.ts              — геометрия + упрощённая физика полёта
  systems/
    AircraftController.ts            — клавиатурный ввод → физика самолёта
    ChaseCamera.ts                     — сглаженная камера от третьего лица
    QualityManager.ts                   — измерение FPS → переключение пресета качества
    FreeCameraController.ts              — БОЛЬШЕ НЕ ИСПОЛЬЗУЕТСЯ Engine'ом, оставлен в проекте
                                            (пригодится для будущего debug/spectator-режима)
  config/Constants.ts                — + AIRCRAFT, QUALITY_TIERS
```

## Известные ограничения (честно, не скрыто)

- Физика полёта упрощённая (см. выше) — не путать с полной аэродинамикой.
- ChaseCamera без свободного обзора (нет "оглянуться назад"/кабины) — только
  фиксированный вид от третьего лица.
- QualityManager переключает только radius/shadows/pixelRatio — резолюцию
  чанка (97) и размер shadow map (1024) в рантайме не меняет, это статичные
  константы. Если понадобится ещё более агрессивная адаптация — следующий шаг.
- Растительности, коллизий с рельефом, HUD-приборной панели (кроме
  отладочной текстовой панели) всё ещё нет.

# Этап 9: полная аэродинамика (угол атаки, срыв потока) + приборная панель

Проверено: `tsc --noEmit` (strict) и `vite build` проходят чисто.

## Полная аэродинамика (`physics/FlightModel.ts`)

Раньше подъёмная сила была грубым приближением "растёт со скоростью вдоль
носа". Теперь `physics/FlightModel.ts` — отдельный чистый физический модуль
(не завязан на рендер/меш), считающий:

- **Угол атаки** — угол между продольной осью самолёта и вектором скорости
  в вертикальной плоскости самолёта (через перевод скорости в body-оси).
- **CL(α)** — коэффициент подъёмной силы: линейный рост до критического угла
  (15°), затем **срыв потока** — CL резко падает к более низкому плато.
- **CD(α)** — паразитное сопротивление + индуктивное (∝CL²) + дополнительное
  при развитом срыве.
- **Направление подъёмной силы** — векторно, как составляющая "верха"
  самолёта, перпендикулярная вектору скорости (не жёстко "вверх по фюзеляжу").
- **Эффективность рулей** теперь падает на малой скорости (∝ скоростному
  напору, т.е. квадратично от скорости) — на рулении/сразу после сваливания
  самолёт слушается управления вяло, как в реальности.

Числа подобраны под параметры, близкие к лёгкому самолёту (масса 900 кг,
крыло 16 м² — почти точно Cessna 172), и **проверены двумя способами**:

1. Аналитически (равновесие сил горизонтального полёта):
   ```
   Скорость сваливания:            ~85 км/ч   (реальная Cessna 172: ~87 км/ч)
   Крейсер при газе 0.5:           ~164 км/ч  (тяга 750Н против нужных 744Н —
                                                 самолёт держит высоту почти сразу)
   Скорость на полном газу:        ~260 км/ч
   ```
2. Численным интегрированием разгона по земле на 30 сек (газ 0.5, без
   ввода тангажа) — самолёт стабильно разгоняется, не взлетает раньше
   времени (нужно "поднять нос" для набора угла атаки, как в реальности),
   нет NaN/дивергенции.

**Честно про оставшиеся упрощения** (без прикрас): постоянная плотность
воздуха (без спада с высотой), одна общая "эффективность рулей" вместо
разных характеристик elevator/aileron/rudder, нет ground effect, нет
обледенения/чисел Рейнольдса/сжимаемости. Это уже настоящая зависимость от
угла атаки со срывом потока — качественно другой уровень, чем "подъёмная
сила ~ скорость" в предыдущей версии, но не сертифицируемый авиасимулятор.

## Приборная панель (`ui/InstrumentPanel.ts`)

Правый нижний угол экрана: SVG-авиагоризонт (небо/земля/линия горизонта,
поворот по крену + сдвиг по тангажу), цифровые показания (скорость км/ч,
высота над рельефом AGL, вертикальная скорость, курс, угол атаки), красный
индикатор "СРЫВ ПОТОКА" при превышении критического угла атаки, вертикальный
индикатор газа. Обновляется каждый кадр (не раз в 0.5с, как отладочная
FPS-панель) — авиагоризонт должен быть плавным.

Высота AGL переиспользует `aircraft.lastGroundHeight`, посчитанный при
проверке столкновения в этом же кадре — не считает высоту рельефа дважды.

# Этап 10: фикс мгновенного срыва потока при старте

Проверено: `tsc --noEmit` (strict) и `vite build` проходят чисто.

## Баг: самолёт стартовал зависшим и мгновенно сваливался

По фидбэку "постоянно стремится упасть, срыв потока постоянно горит,
скорость не набирается, нет реактивности" — все четыре симптома оказались
ОДНОЙ причиной. Самолёт появлялся на высоте 900 м с **нулевой скоростью**
(`velocity = new THREE.Vector3()` — по умолчанию (0,0,0)). При нулевой
скорости подъёмная сила почти нулевая (она растёт с квадратом скорости —
`q = 0.5·ρ·v²`), гравитация тут же утаскивает самолёт вниз, вектор скорости
мгновенно уходит почти вертикально вниз при всё ещё горизонтальном корпусе —
математически это огромный угол атаки, то есть мгновенный срыв потока
буквально с первого кадра. А в развитом срыве подъёмной силы почти нет, а
сопротивление резко выше — разогнаться вперёд физически нечем, самолёт так
и остаётся в этом состоянии (падает с горящим "СРЫВ ПОТОКА", скорость вперёд
не растёт, любой ввод тангажа тонет на фоне и так огромного угла атаки —
отсюда и "нет реактивности").

**Исправлено**: `Aircraft.register()` и `Aircraft.reset()` теперь задают
стартовую скорость `AIRCRAFT.START_SPEED = 48 м/с` вдоль направления носа —
самолёт появляется уже в полёте, а не зависшим. Проверено численным
моделированием 20 секунд полёта БЕЗ единого движения ручкой (руки убраны со
стика): скорость стабильно держится ~173–182 км/ч, угол атаки естественно
устаканивается около 1–1.5° (далеко от критических 15°), лёгкое проседание
~1 м/с (означает лишь слегка недостаточный газ для идеально горизонтального
полёта на этом угле — небольшая коррекция газом/тангажом решает, это
нормальное поведение самолёта, а не авария).
